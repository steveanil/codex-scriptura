# Known Issues & Hardening Backlog

> **Status:** Living document. Findings from the July 2026 full-codebase review.
> Items are removed when fixed (with the fixing commit noted in the PR).
> Severity: **Critical** = wrong/missing scripture data or data loss risk;
> **High** = user-visible malfunction; **Medium** = correctness edge case or
> performance; **Low** = hygiene/consistency.

## Critical - Data Accuracy

### 1. ~~Footnote and study-note text leaks into verse text~~ - **FIXED 2026-07-15**
- **Was:** Both importers stripped XML *tags* but kept the *content* of note elements - 2,514 WEB `<f>` footnotes + 358 `<x>` cross-ref notes and 48 OEB `<note>` bodies were merged into verse text (WEB Gen 1:1 read *"…God The Hebrew word rendered "God" is "אֱלֹהִ֑ים" (Elohim). created…"*).
- **Fix shipped:** `core/xml.ts removeElements()` removes note elements with their content *before* lemma extraction, wj-offset computation, and text stripping, so words-of-Jesus offsets stay aligned. Dexie **v16** clears `verses` + `searchIndexes` for re-seed. Covered by `importers/import-texts.test.ts`. Verified: 0 remaining leaks across all three translations; the 25 WEB + 19 OEB records whose entire content was an omission footnote ("Some manuscripts add…") correctly no longer appear as scripture.
- **Future option (still open):** extract footnotes into their own field/table instead of discarding - translation footnotes are useful study data.

### 2. ~~Bridged verses are silently dropped (WEB)~~ - **FIXED 2026-07-15**
- **Was:** The `<v id="(\d+)"` token regex never matched bridges like `<v id="15-16"/>`, so bridged content was skipped.
- **Fix shipped:** importer parses `id="(\d+)(?:-(\d+))?"`, stores the entry under the first verse number with a new optional `verseEnd` field (`VerseRecord`, `RawVerse`, seed passthrough), and `ReaderPane` renders the label as "15–16". In the current WEB source all 5 bridges turn out to be omission placeholders (footnote-only → correctly empty after fix #1); the support is load-bearing for future translations with real bridged text. Covered by `importers/import-texts.test.ts`.
- ~~Cross-references targeting the *second* verse of a bridge don't resolve~~ - **FIXED 2026-07-15**: `getVerse()` falls back to the bridge record covering the requested verse number (unit-tested).

### 3. ~~No versification validation in the pipeline~~ - **FIXED 2026-07-15**
- **Shipped:**
  1. `validate-texts.ts` pipeline stage (runs automatically at the end of `import:all`; also `pnpm run validate:texts`): checks duplicates, unknown books, empty text, chapter counts vs the canonical table, bridge overlaps, and per-chapter verse gaps. Gaps are classified against `KNOWN_VERSE_GAPS` - a fully source-verified list of text-critical omissions (Acts 8:37 class) and source versification skips (Greek Esther, Prayer of Azariah, Sirach "best authorities" bridges). Handles Apocrypha numbering variants (EpJer as Baruch 6, AddPs as Psalm 151, sparse Greek Esther chapters). Report written to `data/processed/_metadata/text-validation.json`; hard errors fail the pipeline. All three current translations validate clean.
  2. Golden-sample tests (`golden-texts.test.ts`): exact-text assertions for anchor verses per translation plus a leaked-footnote-phrase sweep; they run wherever the pipeline data exists and skip cleanly in CI.
- ~~Documented limitation: omissions of a chapter's *last* verse(s) are undetectable by gap analysis~~ - **closed by #24's trailing-verse check (2026-07-18)**.

### 18. ~~`removeElements()` can delete scripture words and still leaks nested-note text~~ - **FIXED 2026-07-15** *(was a regression in fix #1)*
- **Was:** A self-closing note **with attributes** matched the opening-tag pattern (`[^>]*>` consumed the `/>`) and the non-greedy body extended to the next real closing tag - `foo <note n="a"/> bar <note …>fn</note> baz` imported as `foo  baz`, silently deleting scripture words. Nested same-tag notes also leaked the outer note's tail text into verse text.
- **Fix shipped:** `core/xml.ts removeElements()` rewritten as a depth-aware scanner: self-closing forms (with or without attributes) are never treated as openers, nested same-tag notes are removed in full, and an element left unclosed by the fragment boundary drops the remainder (note content) rather than leaking it. Regression tests for all three cases in `importers/import-texts.test.ts`. Re-import verified byte-identical output for KJV/OEB/WEB (the bugs were latent for current sources) with validation green.

### 19. ~~USFX lettered verse segments are silently dropped~~ - **FIXED 2026-07-15** *(was residual of #2)*
- **Was:** `<v id="15a"/>` didn't match the verse-token regex (closing quote had to follow the digits), so `verseStart` was never set and the segment's content was skipped - relevant for eBible.org translations planned for v0.5.0.
- **Fix shipped:** the token regex accepts lettered ids on either side of a bridge (`15a`, `16a-17`), and segments sharing a verse number merge into one record (text appended, words-of-Jesus offsets shifted, lemma tokens unioned, `verseEnd` maxed) so no duplicate records reach the validator. Covered by `importers/import-texts.test.ts`; current KJV/OEB/WEB output verified unchanged.

## High

### 4. ~~`seedTranslation` never checks the fetch response~~ - **FIXED 2026-07-15**
All seeders now go through `fetchJsonAsset()` (`src/lib/seed.ts`), which checks `res.ok` **and** validates the body parses as JSON - catching SPA hosts that serve `index.html` with HTTP 200 for missing files. Each `seedAll()` step is individually try/caught so one dataset failing never aborts the rest. UI surfacing of failures remains open under #16.

### 5. ~~Chapter-loading race conditions~~ - **FIXED 2026-07-15**
Both `loadChapter` implementations (`ReaderWorkspace.svelte`, `PaneState`) now use an incrementing generation counter and bail before every state assignment when stale - rapid navigation can no longer interleave verses/annotations/enrichment from different chapters.

### 6. ~~Service-worker offline fallback is never pre-cached~~ - **FIXED 2026-07-15**
`service-worker.ts` now caches `prerendered` pages in `ASSETS` and explicitly caches `/index.html` (best-effort - the dev server doesn't serve it). Verified present in the production build output.

### 7. ~~IndexedDB persistence is never requested~~ - **FIXED 2026-07-15**
`+layout.svelte` requests `navigator.storage.persist()` at boot (non-blocking). Settings gained a **Storage** panel showing persistence status (with a manual "Request persistence" retry) and a `storage.estimate()` usage meter.

### 20. ~~Low-confidence person matches persist `bibleDataId` and can attach genealogy edges to the wrong person~~ - **FIXED 2026-07-16**
**Was:** the old enrich script only persisted a link when a name meaning corroborated the match; the extracted `importers/enrich-persons.ts` wrote `bibleDataId` for **all** resolved persons - including 0.80-confidence label-fallback matches - and `import-genealogy.ts buildExactIdMap` consumed them with no confidence filter, so a wrong unique-label match silently gained wrong ancestor/half-sibling edges. **Fix shipped:** `enrich:persons` now persists `bibleDataConfidence` (the stage tier) alongside each link, and `buildExactIdMap` excludes links below 0.85 unless meaning-corroborated (`nameMeaningSource === 'bibledata'` - the old script's guard); a missing confidence field (stale persons.json) is treated as low, not trusted. On current data this excludes 11 uncorroborated links (e.g. `epaphras → Epaphroditus_1`, `claudius_948 → Claudius_2`) - none of which had supplement rows, so genealogy.json is byte-identical: the fix is protection against future data, not a behavior change. Covered by tests in `import-genealogy.test.ts` / `enrich-persons.test.ts`.

### 21. ~~One stray quote in a BibleData CSV silently truncates all following rows~~ - **FIXED 2026-07-16**
**Was:** `core/csv.ts parseCsv` toggled quote mode on any `"` mid-unquoted-field and then swallowed commas **and newlines** until the next quote - one unbalanced interior quote (plausible in `hebrew_label_meaning`-style prose fields) made the rest of the file a single field, silently truncating label/person/relationship parsing. **Fix shipped:** a quote now opens a quoted field only at the *start* of a field (RFC 4180); mid-field quotes are literal text, so a malformed value damages at most its own field. Regression tests in `core/csv.test.ts`.

### 25. ~~Graph "Show all links" lags, breaks rendering, and needs repeated clicks to recover~~ **(FIXED 2026-07-16)**
Root cause confirmed by measurement: all edge `<path>` elements were permanently mounted at opacity 0, each with a 0.18s CSS transition, and with real cross-reference data the ring is a near-complete graph (1,791 edges at the minWeight=5 threshold, out of 2,035 connected pairs). Toggling **Show all links** flipped opacity on ~1,800 Bézier paths at once; the mass style recalc plus paint stall is what read as "everything disappears until I click around." Fix in `src/routes/graph/+page.svelte`: edges are now mounted only when visible (a `drawnEdges` derived), and "show all" renders only the strongest 300 links (`SHOW_ALL_EDGE_CAP`), applying the app's own cap principle. Faint opacity now scales with link weight, and the toolbar count reads "strongest 300 of 1,791 links" so the cap is honest. Selection behavior is unchanged.

### 26. ~~Native dropdowns are unreadable in dark mode (white-on-white)~~ **(FIXED 2026-07-16)**
Root cause as diagnosed: the app themed everything via CSS custom properties but never set the CSS `color-scheme` property, so native `<select>` popups kept the UA light rendering while inheriting light text. The closed controls were also affected because their dark "background" is a nearly transparent white wash (`--color-bg-surface: rgba(255,255,255,0.045)`) over the UA's white base. Fix in `src/app.css`: `color-scheme: dark` on `:root` (dark is the default palette) and `color-scheme: light` inside the `[data-theme="light"]` block, so it tracks the resolved theme automatically through both code paths that set `data-theme` (`+layout.svelte` preferences effect and `toggleTheme`). Audited both select sites (Settings `.select-input`, ReaderWorkspace `.translation-picker` in main nav and per-pane bars); neither hardcodes a conflicting background. Bonus: UA scrollbars and any future date inputs now follow the theme too. Follow-up same day: the popup was still light in testing because Chromium on Linux can ignore `color-scheme` for the `<select>` popup when the control has an author-set background, so `app.css` now also pins `select option` to `--color-bg-elevated` / `--color-text-primary` explicitly. Second follow-up: with `color-scheme` set, Chromium draws its own dark form-control chrome (a heavy light-gray border and arrow) on top of the app styling. Both selects (Settings `.select-input`, ReaderWorkspace `.translation-picker`) now use `appearance: none` with a custom chevron, hover state, and accent focus ring instead. Third follow-up (user screenshot from Brave): the open popup still had a white frame and pale-blue selection highlight because Chromium derives the popup's light/dark rendering from the select's background color, and the translucent `--color-bg-surface` wash (`rgba(255,255,255,0.045)`) reads as white. Firefox doesn't use this heuristic, which is why it looked fine there. Both selects now use new solid `--color-bg-control` / `--color-bg-control-hover` vars (visually identical to the wash over the dark bg), and `select option:checked` gets an accent-tinted highlight.

### 27. ~~Accent color not applied consistently - several UI elements hardcode sky blue~~ **(FIXED 2026-07-16)**
Real root cause, sharper than originally filed: the Settings accent control set only `--color-accent`, but the palette defines an accent *family*. `--color-accent-hover` (which verse numbers and the active nav tab chain off), `--color-accent-subtle`, `--color-search-highlight`, and `--shadow-glow` stayed factory sky blue. Fix: the `+layout.svelte` preferences effect now derives the whole family from `prefs.accentColor` per resolved theme (hover = lighten 35% dark / darken 12% light; subtle, highlight, and glow as alpha washes), via a new tested `src/lib/utils/color.ts` (hexToRgb, withAlpha, lighten, darken). Hardcoded accent blues swept: `.nav-item.active` in `+layout.svelte`, LineageRail's open-tree button, the graph search focus ring (all now `color-mix()` washes over `var(--color-accent)`), and GenealogyTreeModal's range `accent-color`. Deliberately left: the sidebar logo gradient (brand mark, not themed) and the `/search` Greek/Hebrew lexicon chips at `+page.svelte:1046` (semantic language colors, blue = Greek and purple = Hebrew, same category as the OT/NT palette; the "blue in search" complaint was the stale `--color-search-highlight`, now derived).

### 32. ~~Font selects in Settings render blank; chosen fonts had no fallback~~ **(FIXED 2026-07-16)**
Found while retesting #26. The v14 design-refresh migration moved everyone to the bundled defaults (`Instrument Sans` UI, `Newsreader` scripture), but the Settings dropdowns still listed the old font sets (Inter, Crimson Pro, Georgia, ...). The saved value matched no `<option>`, so the select rendered blank until the user manually picked something. Also, Inter and Crimson Pro were offered but never loaded (app.css only imports Instrument Sans, JetBrains Mono, Newsreader), so choosing them silently fell back. Fix in `settings/+page.svelte`: option lists rebuilt around fonts that actually exist (bundled defaults plus common system fonts), and if a saved value isn't listed it's appended as an option so the select always displays the current choice. Bonus in `+layout.svelte`: applied fonts now get a generic fallback appended (`, serif` / `, sans-serif`) so an unavailable font degrades gracefully.

### 33. ~~Cross-reference indicator wraps alone onto its own line~~ **(FIXED 2026-07-17)**
User screenshot (Ruth 1:22): when a verse's last text line is full, the tiny "open" icon with the ref count wrapped by itself onto a new line, looking detached and broken. Root cause is a CSS line-breaking rule, not whitespace: the badge was a `<button>` with `display: inline-flex` containing an `<svg>`, and atomic inlines get a line-break opportunity before them even with no space and even through a word joiner (both were tried and verified insufficient in Chromium; `position: absolute` at static position was also tried and lands unpredictably in wrapped text). Fix in `ReaderPane.svelte`: the xref and quotation badges are now plain `display: inline` spans (`role="button"`, `tabindex`, Enter/Space handlers) with the icons as CSS masks instead of `<svg>`, wrapped in a nowrap `.verse-badges` span glued to the verse html with a leading U+2060. Pure inline text follows character break rules, so the badge now always wraps together with the verse's final word ("...of barley / harvest.↗3"). Verified with headless-browser screenshots on Ruth 1:20–22.

### 34. ~~God has a family tree~~ **(FIXED 2026-07-17)**
User feedback: God shouldn't appear as a person with a family tree. Theographic encodes Luke 3:38 ("Adam, which was the son of God") as literal family columns, giving `god_1324` father-of edges to Adam and Eve - which made God the apex of every ancestry walk. Three-layer fix: (1) the genealogy importer now excludes any edge touching divine IDs (`DIVINE_IDS` in import-genealogy.ts, tested; genealogy.json regenerated 4,481 → 4,479 links and copied to static); (2) Dexie v17 upgrade deletes the two edges from already-seeded databases (verified against a live seeded profile); (3) the entity panel's "Family tree" button now only renders when the person actually has genealogy links, which also removes the dead button from minor figures whose tree would be a single lonely card. Jesus' human-family links (Joseph, Mary, siblings) are untouched - that genealogy is Matthew 1 / Luke 3 content. Bonus while in the panel: person descriptions now render through DictDefinition, so Easton's text there gets the same sense-splitting and tappable refs as the dictionary card (#28).

### 28. ~~Dictionary card: "Search in Bible" drops the word, and Easton's text is a wall~~ **(FIXED 2026-07-17)**
Two halves of the same double-click-lookup feature, both fixed as filed:
- **The button half:** `/search/+page.svelte` now reads `?q=` on mount, prefills, and auto-runs. Single words default to Word Study mode (exhaustive concordance), phrases to Full Text, and an optional `?mode=fulltext|concordance|lexicon` param overrides. Bonus: index-ready callbacks now only re-run the search in Full Text mode instead of unconditionally. Verified in-browser: `/search?q=barley` lands with the query filled, Word Study active, and "38 occurrences in 36 verses" already rendered.
- **The readability half:** new `src/lib/utils/dictionary.ts` (10 tests) splits "(1) … (2) …" numbered senses into blocks and tokenizes scripture refs (abbreviations like "1 Cor. 15:45", verse ranges, and ", 17" continuations resolve through core's `resolveBook`; a lone "(1)" doesn't trigger sense-splitting, and "(1 Cor. 3:2)" isn't mistaken for a marker). New `DictDefinition.svelte` renders each ref as a button wired to the existing `verseHover` preview and reader navigation; used by both the word-lookup card and EntityDetailPanel's Easton's sections. Verified in-browser: the "barley" card renders 10 tappable refs, hovering shows the verse preview, and clicking navigates the reader to Exodus 9.

### 22. ~~Text-validation stage blind spots~~ - **FIXED 2026-07-18** *(gaps in fix #3 - branch review 2026-07-15)*
All three blind spots closed in `core/validate-texts.ts` (unit-tested; all three translations re-validated clean with identical expected-omission counts):
- **Empty file passed:** `validateTranslation` now emits a hard error on zero verse records, so an empty `*-verses.json` fails the pipeline instead of reporting "ok - 0 verses". (The CLI also now derives the translation ID from the filename unconditionally - the old fallback labeled an empty file's report with the raw filename.)
- **`v.chapter` never validated:** chapter numbers are now checked like verse numbers (integer, ≥ 1); chapter-0/negative/fractional records are hard errors and never enter coverage.
- **`KNOWN_VERSE_GAPS` was translation-agnostic:** now a per-translation map keyed by translation ID. The keying was verified against the last validation report and is *tighter* than the issue asked: WEB's source retains 15 of the 19 NA/UBS disputed NT verses, so its set carries only the 4 it actually omits (Luke 17:36, Acts 8:37, 15:34, 24:7) - previously a WEB importer bug dropping Matt 17:21 would also have stayed green. A translation with no entry gets an empty set (every gap warns until source-verified).

### 24. ~~Trailing-verse omissions are undetectable by gap analysis~~ - **FIXED 2026-07-18** *(documented limitation of fix #3, promoted to its own item so it stays actionable)*
- **Reference table shipped:** `core/kjv-versification.ts` - per-chapter verse counts for all 81 KJV books, generated by `pnpm run generate:versification` from the raw KJV OSIS verse milestones (deliberately *not* from importer output, so an importer bug can't bake itself into the reference). The generator cross-checks independent facts before writing: 31,102 verses in the 66-book canon, spot counts (Gen 1 = 31, Ps 119 = 176, …), canon chapter bounds, and a full diff against the current processed KJV (agreed on every chapter). The table is checked in, so validation runs without raw sources.
- **Trailing check in `validateTranslation`:** each present chapter's max verse is compared against the reference; shortfalls are classified through the same per-translation `KNOWN_VERSE_GAPS` sets (#22), overruns warn. Chapters absent from the table (WEB-only Apocrypha like 3Macc) are skipped.
- **First run found exactly the predicted class, all source-verified:** WEB Sir 20:32 and Sir 23:28 are "omitted by the best authorities" footnote-only verses (added to WEB's known gaps); WEB relocates the Romans doxology to 14:24–26 and ends Romans 16 at v24 (16:25–27 added as expected omissions); OEB 3John 15, OEB Rev 12:18, WEB Sir 44:23, and WEB PrMan 1:15 are legitimate modern-versification numbering differences with real text - captured in a new source-verified `KNOWN_VERSE_COUNT_VARIANTS` map that overrides the reference count per translation/chapter. All three translations validate clean (0 warnings; WEB expected omissions 46 → 51).
- The multi-versification mapping layer for non-KJV-versified translations remains future work (v0.5.0+ eBible.org translations), but `KNOWN_VERSE_COUNT_VARIANTS` covers per-chapter deviations until then.

### 35. ~~Preferences and nav-history never actually persisted - every Dexie save threw `DataCloneError`~~ - **FIXED 2026-07-18** *(found verifying #14; pre-existing, confirmed on pre-refactor code)*

### 16. ~~Silent failure everywhere on first run~~ - **FIXED 2026-07-18** *(error surfacing + lightweight progress; the polished onboarding screen stays a v0.9.0 roadmap item)*

A schema upgrade blocked by another tab (`db.on('blocked')`) shows "waiting for another tab to close" instead of hanging silently.
